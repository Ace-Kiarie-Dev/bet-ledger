## 2026-07-09 — Google Sign-In only for auth

**Context:** BetLedger's original spec included phone auth and possibly email/password.
**Decision:** Google Sign-In is the only auth method. No email/password, no phone auth.
**Why:** Simpler UX for the Kenyan market (almost everyone has a Google account via Android), no password management burden, no OTP costs, works with Firebase's ID token exchange out of the box.
**Trade-off accepted:** Users without a Google account can't sign in. Deemed acceptable given target market.

## 2026-07-09 — Firestore profile path: `profile/data`, not `meta/profile`

**Context:** Original `CLAUDE.md` schema used `users/{uid}/meta/profile`. Session 7 rebuild questioned whether this was right.
**Decision:** Use `users/{uid}/profile/data` (sub-collection: profile, doc: data).
**Why:** Consistent with `stats/data`. Simpler mental model — every sub-thing under a user is `<collection>/data`. `meta` was doing no work.
**Migration status:** Any orphaned old `meta/profile` docs in production Firestore are harmless (nothing reads them).

## 2026-07-09 — `createUserProfile` accepts the full Firebase user object

**Context:** Session 7 discovered that `DashboardScreen` and (planned) `AuthScreen` were both going to need to auto-create profile documents. Original signature was `createUserProfile(uid, { username })`, requiring each caller to derive the username itself.
**Decision:** Signature is now `createUserProfile(user)` where `user` is the Firebase user object. The helper derives the username internally via priority chain: `user.displayName → user.email prefix → user.phoneNumber → 'Player'`.
**Why:** Sanitization lives in one place. Callers can't accidentally pass `undefined` and break Firestore. AuthScreen and DashboardScreen call it identically.

## 2026-07-15 — Account deletion via Cloud Function, not a Firestore rules carve-out

**Context:** BetLedger needs an account deletion path. The obvious client-side implementation — sign in, then run a batch of `deleteDoc` calls against `users/{uid}` and its subcollections directly from the app — was considered and rejected. `firestore.rules` currently locks settled bets against deletion (`allow delete: if isOwner(userId) && !isSettled(resource.data)`) by design, so a bet that's already been won or lost is a permanent record; that's the entire point of the "honest ledger" guarantee. A client-side deletion path would need a rules carve-out letting the owner delete settled bets too (at minimum for their own account-wipe case), which weakens that guarantee for any other write path that could piggyback on the same carve-out. This mirrors the earlier Leaderboard decision (see "Leaderboard: no cross-user data until Firestore rules + write-strategy are decided" above) — client-side writes/deletes get rejected whenever they'd require loosening rules that exist for a specific integrity reason, in favor of a server-side path that goes around rules entirely rather than through a hole in them.
**Decision:** Account deletion runs through a new callable Cloud Function, `deleteAccount` (`functions/index.js`, region `europe-west1` to match Firestore), using the Admin SDK. Admin SDK access bypasses Firestore rules by design, so `firestore.rules` needed zero changes — the outcome-lock rule stays exactly as-is for every other path into Firestore. The function verifies `context.auth` and uses `context.auth.uid` as the delete target; it never trusts a uid from the client payload. It walks `users/{uid}` and every subcollection under it via `listCollections()` (not a hardcoded `[bets, stats]` list, so a subcollection added in a future session isn't silently skipped) and deletes recursively in batches of 400.
**Deletion order:** Firestore data is wiped first, the Firebase Auth user is deleted last. If the Firestore wipe throws partway through, the Auth account is still alive and the user can just retry — the alternative order risks an Auth-deleted account with orphaned Firestore data that's no longer reachable by anyone (the uid that owned it no longer resolves to a signed-in session).
**Client side:** `app/utils/deleteAccount.js` re-authenticates via `GoogleSignin.signIn()` (Google Sign-In is the only auth method, per the 2026-07-09 decision above) before calling the function, since a stale session throws `auth/requires-recent-login`. On success it also calls `signOut(auth)` locally so the app routes back to Auth immediately rather than sitting on a signed-in screen for an account that no longer exists server-side.
**functions/ did not exist yet:** created it this session — `functions/index.js`, `functions/package.json` (`firebase-admin ^12.1.0`, `firebase-functions ^5.0.1`, Node 20), and added a `"functions": { "source": "functions" }` block to root `firebase.json` (not `firestore.rules` — a different file, needed so `firebase deploy --only functions` can find the source).
**Judgment call flagged:** the style guide has no documented "irreversible destructive action" button pattern beyond the `coral`/`COLORS.loss` token — the only precedent in code is the outlined pill used for Sign Out (Settings, Profile). Delete Account reuses `COLORS.loss` but as a **filled** pill (vs. Sign Out's outline) to visually rank it as more severe/less recoverable than signing out; text color reuses `COLORS.background` for contrast, following the same dark-on-light construction as `onPrimary` since no `onLoss`/`onCoral` token exists. Worth formalizing in the style guide if more destructive actions get added later.
**Still outstanding (not done this session):** a web-accessible account-deletion request page (can live on the privacy policy page — Play Store's Data Safety section requires this even though in-app deletion now exists), and deployment (`firebase deploy --only functions`) hasn't been run yet.

## 2026-07-15 — AuthScreen: two-path Google Sign-In (cached "Continue as X" + account switch)

**Context:** With Google Sign-In as the only auth method (see 2026-07-09 above), the native `@react-native-google-signin` layer caches the last-used account locally. `GoogleSignin.signIn()` reuses that cache silently when one exists, which is normally desirable (fast repeat sign-in) but meant there was no way for a user to switch to a different Google account — `signIn()` just kept returning the same cached one, with no picker ever appearing. Redesigned per an attached Stitch mockup (`assets/Sign Up Options.png`) to a two-path layout that fixes this.
**Decision:**
- On mount, `AuthScreen` checks `GoogleSignin.hasPreviousSignIn()` / `GoogleSignin.getCurrentUser()` (both synchronous native-cache reads, confirmed from the installed `@react-native-google-signin/google-signin` v16 source — no network call, no loading state needed).
- If a cached account exists: two equal-width pill buttons render side by side — left is "Continue as {name}" (silent re-auth via `GoogleSignin.signInSilently()`, no picker), right is a plain "Google" button.
- **The actual fix for "stuck on one account":** the right ("Google") button calls `GoogleSignin.signOut()` *first* to clear the native cache, *then* `GoogleSignin.signIn()`. Calling `signIn()` alone never surfaces the picker while a cached session exists — clearing the cache first is what forces it.
- If no cached account exists, only a single centered "Sign in with Google" button renders — no empty/disabled left button.
**Why this matters for Sign Out:** the native cache this screen depends on is only useful if Sign Out doesn't wipe it. **Confirmed** (not changed — no prior session had added it) that both Sign Out handlers (`SettingsScreen.js`, `ProfileScreen.js`) call only `signOut(auth)` — the Firebase sign-out — and never `GoogleSignin.signOut()`. If either had called `GoogleSignin.signOut()` on every sign-out, the native cache would be cleared every time and the "Continue as X" fast path would never have anything to show, defeating the point of this screen.
**Judgment calls flagged:**
- The mockup shows a "Skip" link top-left of the hero. Per explicit instruction this was omitted rather than added — AuthScreen had no Skip button before this change, and the instruction said not to add one back.
- After `GoogleSignin.signOut()` in the account-switch handler, `cachedUser` state is cleared immediately (before the picker even resolves) — so if the user cancels the picker, the screen correctly falls back to the single-button layout instead of showing a now-stale "Continue as X" for a session that no longer exists in the native cache.
- The cached-user label truncates by name first, then email (to 17 chars + ellipsis) if no display name is set, per spec; falls back to a generic "Continue" label in the (unexpected) case where the cached `User` object has neither.

## 2026-07-15 — Account deletion moved to client-side; `bets` delete-lock relaxed (supersedes part of the earlier 2026-07-15 Cloud Function entry)

**Context:** The `deleteAccount` Cloud Function (logged earlier today — "Account deletion via Cloud Function, not a Firestore rules carve-out") was written and correctly deployable in principle, but deployment was blocked: 2nd-gen Cloud Functions require the Blaze (pay-as-you-go) plan, and this project is on Spark. **Decision made this session: defer upgrading to Blaze entirely until a cross-user feature (the leaderboard) actually requires server-side enforcement.** Nothing about account deletion justifies a billing-plan change on its own.
**Decision:** Account deletion now runs fully client-side (`app/utils/deleteAccount.js`), through `firestore.rules` directly rather than around it via the Admin SDK. This required relaxing `firestore.rules`'s `bets` delete condition from `isOwner(userId) && !isSettled(resource.data)` to just `isOwner(userId)`. The **update** rule (which enforces the actual outcome-lock — settled bets can't have `outcome`/`stake`/`odds`/`market`/`selection` edited) is untouched and fully intact; only whole-record deletion was relaxed, and only so a user can delete their own entire account.
**The trade-off, stated plainly:** a modified client can now delete individual settled bets, not just whole accounts. It cannot edit a settled bet's outcome — that guarantee is unchanged. Someone could use a hacked client to selectively erase specific losing (or winning) bets from their own history while keeping others.
**Why this is acceptable right now:** there is no cross-user feature live. Leaderboard (see 2026-07-10 "no cross-user data until Firestore rules + write-strategy are decided") still only ever shows the current user their own computed streak — nobody else's integrity depends on any other user's data being tamper-proof. The only person a tampered delete could mislead is the tamperer themselves, viewing their own history.
**RE-TIGHTENING TRIGGER (explicit):** before ANY cross-user feature ships — the leaderboard going live with real cross-user data, verified badges, or anything else where one user's data integrity is trusted by another user or by the app on their behalf — the `bets` delete rule must be restored to `isOwner(userId) && !isSettled(resource.data)`, and account deletion must move back to the parked Cloud Function (`functions/index.js` — left in place, unwired, with a comment marking it parked for exactly this).
**Data-trust caveat:** any bet created or deleted while the delete-lock is loose is unverifiable for future cross-user purposes — there's no way to retroactively prove a given user's historical bet list wasn't selectively edited during this window. When the leaderboard is built, clean-streak/budget-streak computations should count from the re-tightening date forward, not from account creation, for any account that existed during the loose-rules period.
**Implementation notes:**
- The client SDK has no `listCollections()` (Admin-only), so `app/utils/deleteAccount.js` hardcodes the two subcollections the app actually writes — `profile` and `bets` — derived from `app/utils/storage.js`, with a comment marking it as the place to update when a new subcollection is added. `stats` is excluded: it has no client-side writer (`getStats` has no matching setter) and `firestore.rules` blocks all writes to it (`allow write: if false`) regardless.
- There is no top-level `users/{uid}` document to delete — nothing ever writes one directly (profile data lives at `users/{uid}/profile/data`), and `firestore.rules` grants no access to that bare path (it falls through to the deny-all catch-all). Deletion is scoped to subcollection contents only.
- Deletion order unchanged from the Cloud Function version: Firestore data first, Auth user (`deleteUser(auth.currentUser)`) last, so a failed Firestore wipe leaves a live, retryable Auth account rather than an orphaned one. Retries are safe — deleting an already-empty subcollection is a no-op.
- `SettingsScreen.js` required zero changes — verified, not assumed: `deleteAccount`'s export name, call signature, and generic `err.message` error handling are all unchanged.
- Deployed: `firebase deploy --only firestore:rules` (works on Spark — only Functions requires Blaze). **Still needed: on-device verification** — sign in with a test account that has settled bets, run Delete Account from Settings, confirm the full wipe succeeds and the app routes back to Auth.
- **Follow-up fix (same day):** the success path of `deleteAccount()` now also calls `GoogleSignin.signOut()` (after `deleteUser()` succeeds, alongside `signOut(auth)`), clearing the native Google session cache. Without this, AuthScreen would still offer "Continue as {deleted user}" after a deliberate deletion, silently re-registering a new account under the stale cached identity on tap. Regular Sign Out is intentionally unchanged — it still preserves the native cache for the "Continue as X" fast path (see the AuthScreen two-path decision above); only account deletion clears it.

## 2026-07-17 — Notification preferences: read-time default, not a migration (defaults gap fix)

**Context:** A wiring audit found that `createUserProfile` (`app/utils/storage.js`) never wrote a `notifications` field, while `SettingsScreen` displayed all three toggles as ON via a local `DEFAULT_NOTIFICATIONS` fallback that existed only in component state, never in Firestore. `AddBetScreen`'s scheduling gate (`profile?.notifications?.betReminders`) read straight from the Firestore-backed profile, so for any user who had never opened Settings, that check evaluated `undefined` → falsy → `scheduleMatchReminder` silently skipped. Same for `scheduleStreakAlerts`/`schedulePerformanceUpdates`, which were (and remain) only ever invoked from inside the Settings toggle's `onChange` handler — nothing scheduled them on profile creation or app launch. Net effect: every fresh install showed three toggles that read as "on" while zero notifications were actually scheduled, silently, with no error anywhere in the chain — the toggle state was simply disconnected from reality.
**Decision:** `createUserProfile` now writes `notifications: { betReminders: true, streakAlerts: true, performanceUpdates: true }` explicitly at account creation. For profiles that already exist without the field, `getUserProfile` merges `DEFAULT_NOTIFICATIONS` into `data.notifications` on every read (`{ ...DEFAULT_NOTIFICATIONS, ...(data.notifications || {}) }`) rather than running a one-time migration script to backfill existing Firestore documents.
**Why read-time merge over a migration:** every single read of a profile already goes through `getUserProfile` — there's no other code path that touches this document — so a read-time merge closes the gap for 100% of existing profiles immediately, with zero deploy-ordering risk (no "migration ran before/after this app version shipped" race) and no throwaway one-off script to write, run once, and delete. The only cost is that the Firestore document itself stays field-absent for pre-fix users until they explicitly flip a toggle (which persists the full 3-key object via `persistProfile`) — acceptable since nothing ever reads the raw document directly; everything goes through this same merge.
**Related fix, same session:** `SettingsScreen`'s local `DEFAULT_NOTIFICATIONS` fallback and `ToggleRow`'s `value === undefined ? true : value` fake-default were both removed — the toggle now factors in live OS notification permission too (a preference of "on" with revoked permission is the same class of lie this fix was written to close; see the permission-timing decision below). See `docs/sessions/DECISIONS.md` "Notifications reminders are unconditional" (2026-07-11) for the related, still-standing decision that the recurring reminders don't check live pending-bet state — this fix is about the toggle *defaults*, not that limitation.

## 2026-07-17 — Settle digest: foreground-reconciled one-shot notification, not a background task

**Context:** Kickoff reminders (scheduled at a bet's exact `matchTime`) already existed. A second, generic "you have bets to settle" nudge at 9am local time was requested — one that reflects whether anything is actually still pending, and disappears entirely once nothing is. The technically "correct" way to do this is a background task (`expo-task-manager`) that wakes up, queries Firestore, and recomputes the notification content/schedule independent of whether the app is open. That approach was explicitly rejected.
**Decision:** The settle digest (`scheduleSettleDigest`/`cancelSettleDigest` in `app/utils/notifications.js`) is a single one-shot notification (fixed identifier `settle-digest`, so rescheduling always replaces rather than duplicates) targeting the next local 9am strictly after "now," recomputed fresh on every call. It is rescheduled — not by a background task, but by a foreground orchestration layer (`app/utils/notificationSync.js`'s `reconcileNotifications`) — at four points: (1) app open / auth resolving (`AppNavigator.js`), (2) app foreground-resume via `AppState`'s `'active'` event (`AppNavigator.js`), (3) right after a bet is saved (`AddBetScreen.js`), and (4) right after a bet's outcome is settled to win/loss (`OutcomeUpdateSheet.js`, added same day as a follow-up — see below). Each reconcile queries current bets, checks for any `outcome === 'pending'`, and schedules if found / cancels if not — so the digest can never exist for a user with nothing pending, but only as of the last time reconcile last ran.
**Why background-task computation was rejected — specifically:** `expo-task-manager`'s background fetch relies on the OS actually waking the app process on schedule, and this is unreliable in exactly the way that matters for this app's audience. BetLedger targets Kenyan users, where Xiaomi (MIUI), Oppo (ColorOS), and Samsung (One UI) collectively dominate the Android install base far more than in markets where most background-task documentation and testing assumptions originate. All three of those OEM skins ship aggressive battery-management layers that kill or throttle background app processes by default — MIUI's "battery saver" and app-hibernation behavior in particular is notorious for silently preventing background JS execution regardless of what the app declares in its manifest, and getting a user to manually exempt an app requires OEM-specific settings screens most users never find unprompted. A background task that "mostly doesn't run" on the dominant hardware in the target market is worse than no background task: it would make the feature's actual reliability invisible and undebuggable (it's not an error, it just silently didn't fire), and there's no way to build UI honesty around a scheduling mechanism whose success rate the app itself can't observe. The foreground-reconcile approach has a known, statable limitation instead (below) rather than an OEM-dependent unknown one.
**Settling-the-last-bet case: resolved same day.** The original version of this entry listed "settling the last pending bet, then backgrounding before 9am" as an accepted limitation alongside the timezone caveat. That was wrong to accept — it wasn't reasoned out, it was just missing from the initial trigger list. `OutcomeUpdateSheet.js`'s `handleSave` already called `cancelMatchReminder` when an outcome was set to win/loss; it now also fires `reconcileNotifications(userId)` (fire-and-forget, same non-blocking pattern as AddBetScreen) right after `updateBet` succeeds. Settling is exactly the moment pending-state changes, so reconciling then — not just on next app-open — means the digest is cancelled immediately if that was the last pending bet, within the same session, no reopen required.
**Known limitation that still stands:** because the digest is a one-shot `DATE`-type trigger computed from the device's current local time, it does not self-adjust if the device's timezone changes while the app is backgrounded and not reopened before the originally-scheduled instant fires — it will fire at the old-timezone-equivalent moment, not 9am in the new zone. It self-corrects the next time the app is foregrounded (reconcile recomputes "next 9am" against current local time), so the exposure window is at most one mistimed firing, never a persistent drift. This one is a genuine trade-off of the no-background-task decision above (nothing computes timezone changes while backgrounded, by design) — unlike the settling case above, there's no in-app event to hook to correct it sooner.
**Judgment calls made and flagged:**
- **Gated behind the existing `betReminders` toggle**, not a new fourth toggle — the digest is conceptually the same concern as the kickoff reminder (settling bets), and AddBetScreen already gates the kickoff reminder on this same key, so one label covers both consistently. Reconsider a dedicated toggle if user feedback suggests people want kickoff pings without the digest or vice versa.
- **"Pending" = `bet.outcome === 'pending'`, not gated on whether `matchTime` has already passed.** The spec text was ambiguous between "remind about bets that are ready to settle" (matchTime passed) and "remind about any bet that doesn't have a final outcome yet" (the literal field state). The simpler, literal reading was chosen: it matches the generic, no-live-count copy ("You have bets to settle") and keeps the reconcile logic a single flat query with no time-window edge cases.
- **"9am the following morning" implemented as "next 9am local strictly after now"** (today's 9am if not yet passed, otherwise tomorrow's), used identically for both the bet-creation trigger and the app-open/resume reconcile, rather than hardcoding "always the calendar day after bet creation." This is what makes the reconcile-on-resume path (which needed some target time, not specified as one-off in the request) behave sensibly, and it still reads as "the following morning" for the common case (bet added in the afternoon/evening).

## 2026-07-17 — Kickoff reminder removed; settle digest stays (supersedes parts of the settle-digest entry above)

**Context:** On-device testing on a Tecno Camon 30S (Android 15, HiOS/Transsion) showed the per-bet kickoff reminder (`scheduleMatchReminder`, scheduled for the bet's exact `matchTime` via `SchedulableTriggerInputTypes.DATE`) did not fire while the app was backgrounded — it only appeared once the app was reopened, at which point it's moot (the user is already looking at the app). Root cause: `expo-notifications`' exact-time triggers rely on Android's exact-alarm scheduling path, which requires the `SCHEDULE_EXACT_ALARM` permission on Android 12+ (API 31+). `app.config.js` never declared it.
**Decision:** The kickoff reminder feature is removed outright — `scheduleMatchReminder`/`cancelMatchReminder` deleted from `app/utils/notifications.js`, the scheduling call and `reminderNotificationId` persistence deleted from `AddBetScreen.js`, the cancel-on-settle call deleted from `OutcomeUpdateSheet.js`. `SCHEDULE_EXACT_ALARM` was **not** added to `app.config.js` to unblock it. `reminderNotificationId` is left alone on any existing bet documents that already have it (no migration) — nothing in the codebase reads that field anymore (verified: `OutcomeUpdateSheet.js` was the only reader, and its read was deleted in this same change); it's now inert leftover data, harmless to leave.
**Why patch (add the permission) was rejected, not just "didn't get to it":**
- **Play Store stance:** Google Play's policy on exact alarms discourages their use and specifically flags apps that declare `SCHEDULE_EXACT_ALARM` without a clear, justified use case tied to user-facing exact-time functionality (alarm clocks, calendar events) — a bet-kickoff nudge is a marginal fit for that bar and adds review risk for a feature that, per the second point below, likely wouldn't have worked reliably anyway.
- **HiOS/Transsion (and MIUI/ColorOS/One UI generally) battery management:** even with the permission granted, Transsion's HiOS ships the same class of aggressive background/battery restriction already documented in the settle-digest entry above (Xiaomi/Oppo/Samsung reasoning) — Transsion (Tecno/Infinix/itel) is itself a dominant brand in the Kenyan Android market, arguably more so than the three OEMs named in that earlier entry. Getting the exact-alarm permission granted does not exempt an app from a separate, OEM-controlled decision to defer or kill the process anyway. Fixing the permission gap alone was not confidently expected to fix the observed on-device failure.
- **The digest didn't have this specific failure mode confirmed, and doesn't need exact delivery:** unlike the kickoff reminder — where lateness defeats the entire point (the nudge is "the match just started," so appearing hours later or only on next app-open is functionally useless) — the settle digest's content ("you have bets to settle") is equally true whenever it arrives. That tolerance for drift is why it was kept rather than removed alongside the kickoff reminder.
**Open technical question, flagged rather than assumed away:** the settle digest uses the exact same `SchedulableTriggerInputTypes.DATE` trigger primitive the kickoff reminder used. Tolerance for late delivery only helps if the underlying Android failure mode is "fires late" (inexact alarm, deferred by Doze) rather than "never registers as a background alarm at all without the exact-alarm permission" (which is what the kickoff reminder's symptom — silent until reopen — is at least consistent with). This was not isolated in testing; only the kickoff reminder was diagnosed. **The digest has not been confirmed to survive backgrounding on the same class of hardware** — it's kept on the reasoning that its tolerance makes it more likely to be fine, not on direct evidence that it is. Recommend an on-device background-delivery check for the digest specifically (Tecno Camon 30S or similar HiOS device, app backgrounded well before 9am, screen off) before treating it as reliable.
**Revival path (this is a deferral, not a permanent removal):** a server-driven push notification via a Cloud Function (Firebase Cloud Messaging) would solve the kickoff-reminder problem properly — FCM delivery is server-initiated and goes through Google Play Services' push channel rather than local `AlarmManager` scheduling, so it doesn't need `SCHEDULE_EXACT_ALARM` and is materially less exposed to OEM battery-management interference (high-priority FCM messages have OS-level delivery guarantees local alarms don't). This requires a Cloud Function (to compute "which bets just kicked off" server-side and push to the right user) and therefore the Blaze plan — the same blocker already tracked for the leaderboard's write-strategy decision (see "Leaderboard: no cross-user data until Firestore rules + write-strategy are decided," 2026-07-10, and "Account deletion moved to client-side," 2026-07-15, both blocked/shaped by the Spark-plan constraint). **Unlock condition:** revisit the kickoff reminder via FCM + Cloud Function when Blaze is adopted for the leaderboard — don't build it standalone before then.
**Supersedes, from "Settle digest: foreground-reconciled one-shot notification, not a background task" (2026-07-17, above):**
- That entry's "Gated behind the existing `betReminders` toggle" judgment call justified sharing the toggle partly because "AddBetScreen already gates the kickoff reminder on this same key." That reasoning is now void — the kickoff reminder no longer exists. The toggle-sharing decision itself is unchanged (still no new toggle added), but on its own merits now: `betReminders` also already gates the pre-existing, unconditional daily 8pm reminder (`scheduleBetReminders`/`cancelBetReminders`, see "Notification reminders are unconditional, not state-aware," 2026-07-11) — the digest is the *second* thing sharing this toggle, not something that inherited a slot vacated by the kickoff reminder.
- That entry's four reconcile trigger points (app open, app resume, bet created, bet settled) are unaffected by this removal — `reconcileNotifications` never depended on `scheduleMatchReminder`/`cancelMatchReminder`. No change needed there.
**Still open, not resolved by this change:** `SettingsScreen`'s "Bet Reminders" toggle label/description was written when the toggle's mental model was closer to "reminders about your bets broadly." It now (and, per the correction above, already did before this change) governs two things — a daily unconditional nudge and a conditional settle digest — neither of which is about kickoff timing. Whether the label still reads correctly was flagged for the user to decide, not changed unilaterally in this session; see that session's response for proposed wording options.

## 2026-07-17 — Notifications removed from v1 entirely (supersedes the settle-digest and defaults-gap entries above)

**Context:** A temporary debug pass (`__DEV__`-gated button in Settings, console logging at each reconcile gate and at schedule time) was added to diagnose why the settle digest wasn't firing. Before that instrumentation was even needed to isolate *which* gate was closing, on-device testing (Tecno Camon 30S, Android 15, HiOS/Transsion) already showed the more fundamental problem: **the settle digest didn't fire at all** — not in the background, not on reopen. This is a worse failure than the kickoff reminder's (which at least fired once the app was reopened). Two different notification features, two different failure modes, same root-cause family: local scheduling via `expo-notifications` is not reliable in the background on the hardware that dominates BetLedger's target market. Likely contributors, per the kickoff-reminder investigation earlier this session: Android 12+'s `SCHEDULE_EXACT_ALARM` requirement (never declared, deliberately — see the kickoff-reminder removal entry above) and Transsion's HiOS battery management, both bearing specifically on Tecno/Infinix/itel-class devices that are common in Kenya.
**Decision:** The entire notifications feature surface is removed from v1 — not just the two mechanisms already found broken (kickoff reminder, settle digest), but the two that were never independently tested either (the daily 8pm "Bet Reminders" nudge and the 9am "Streak Alerts" nudge use the exact same local-scheduling mechanism and have no reason to behave differently on the same hardware). Concretely, removed from the live app:
- `SettingsScreen.js`: the entire Notifications section (all three toggles — Bet Reminders, Streak Alerts, Performance Updates), the `ToggleRow` component (now unused), the permission-denied explanatory note, and the permission request/re-request handling that existed only to serve those toggles (`requestNotificationPermission` call in the toggle handler, `Linking.openSettings()` fallback, `permissionGranted` state).
- `AddBetScreen.js` and `OutcomeUpdateSheet.js`: the `reconcileNotifications` calls added earlier this session.
- `AppNavigator.js`: the first-authenticated-screen permission request and the `AppState`-driven reconcile-on-resume listener.
- All four `TEMP DEBUG — remove before build` instrumentation sites added earlier this session (two in `notifications.js`, one in `notificationSync.js`, one in `SettingsScreen.js`) — diagnosis is no longer needed; the answer is "it doesn't work," not "which gate is closing."
**Why removal, not "ship it broken but hidden":** the entire reason the defaults-gap fix (see "Notification preferences: read-time default, not a migration," above) existed was to stop the app from showing a toggle in the "on" position while nothing was actually scheduled — that was identified as dishonest UI and fixed. Shipping v1 with working toggles wired to a delivery mechanism now confirmed broken on real, representative hardware is the exact same dishonesty, just moved from "the default is wrong" to "the whole feature doesn't do what it claims" — a bigger version of the same problem, not a different one. Better to have no notifications section at all than one that silently doesn't work for a meaningful share of the target market.
**Supersedes:** "Settle digest: foreground-reconciled one-shot notification, not a background task" (2026-07-17) and "Notification preferences: read-time default, not a migration (defaults gap fix)" (2026-07-17), both earlier in this same session. Neither is wrong as *reasoning given what was known at the time* — the defaults-gap fix was a correct fix to a real bug, and the digest's design (foreground-reconciled, permission-aware, tolerant of drift) was a reasonable answer to the background-task-reliability problem. What supersedes them is the on-device result: the delivery mechanism itself doesn't work reliably enough on this hardware for either fix to matter in practice. Kept, not deleted, for exactly that reason — they're the correct pattern for whenever local scheduling *is* replaced with something that actually delivers.
**Not touched, and why:**
- `storage.js` — `createUserProfile` still writes the `notifications` field, and `getUserProfile` still merges `DEFAULT_NOTIFICATIONS` on read. Left in place: harmless (nothing reads or displays it now), and the field will be needed again verbatim when the FCM rebuild lands — no reason to delete a working piece of schema just because the UI reading it is temporarily gone.
- `app.config.js` (`expo-notifications` plugin, `android.permissions: ['NOTIFICATIONS']`) and `package.json` (`expo-notifications` dependency) — left declared, not stripped. Reasoning: (1) the FCM-based rebuild will very likely still depend on `expo-notifications` itself — Expo's library handles both local scheduling and remote push token registration/handling through the same API, so this isn't a dependency that's actually going away, just a usage pattern that is; stripping now and re-adding later would mean two native rebuilds for zero net benefit. (2) `POST_NOTIFICATIONS` (what `permissions: ['NOTIFICATIONS']` maps to on Android 13+) is not a sensitive/dangerous permission category — Play Console doesn't require a declaration form or usage justification for it the way it does for SMS, Call Log, Accessibility Service, or precise location; an unused-but-declared instance of it is low-risk. (3) Stripping would force a native rebuild just to remove something dormant, which is itself a cost with no corresponding benefit given point 1. Flagged as a judgment call, not treated as obviously correct — revisit if Play Console review ever actually raises it.
**Unlock condition (updated — now covers three deferred things, not one):** rebuild notifications via FCM + Cloud Function when the Blaze plan is adopted for the leaderboard. This is the *same* trigger already tracked for two other deferred pieces of work: the kickoff reminder's revival path (this session, above) and the parked `deleteAccount` Cloud Function (`functions/index.js`, parked per "Account deletion moved to client-side," 2026-07-15). With three independent features now gated on the same Blaze-adoption event, the case for treating that as one deliberate migration — provision Blaze, stand up Cloud Functions for all three at once, re-tighten `firestore.rules` where needed (see the account-deletion entry's re-tightening trigger) — is stronger than doing any of them piecemeal. Whoever picks up Blaze adoption should treat this as a three-item checklist, not just "the leaderboard needs it."
